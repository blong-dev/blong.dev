524K Context on a Single RTX 5090: Turbo3 KV Compression for Qwen3.5-27B

3 parallel inference slots at 55 tok/s decode, with the VRAM math to prove it.

Three parallel inference slots. Each holding 174,848 tokens of context. Decoding at 55 tokens per second. On a single RTX 5090.

The trick: a 3.5-bit KV cache that fits 524K tokens of total context in the VRAM where FP16 can barely fit one slot at 128K.

This is part of Gnosis, a personal intelligence system that ingests 83 public data sources into a knowledge graph and uses AI agents for research synthesis. The agents need long context windows – not for chatting, but for digesting legislative filings, SEC disclosures, and scientific papers in a single pass. This article covers the inference layer specifically: one GPU, one model, one compression technique, and what the benchmarks actually say.

All benchmark data, scripts, and configuration are provided.

Why Qwen3.5-27B

The 27B dense model is the sweet spot for a 32 GB card at Q6_K quantization – large enough to be genuinely useful, small enough to leave VRAM headroom for KV cache.

The architecture matters here. Qwen3.5-27B is a hybrid: 16 layers use standard grouped-query attention (GQA) with KV cache, and 48 layers use GatedDeltaNet – a linear attention variant that stores persistent recurrent state instead of a KV cache. This means turbo3 only needs to compress 16 layers of KV, not 64. The recurrent state (RS buffer) is a fixed 449 MiB regardless of context length.

This architectural detail is why the compression works so well. You’re only paying the KV tax on a quarter of the layers.

Turbo3: The KV Cache Problem and the Fix

The VRAM Budget at FP16

The RTX 5090 has 32,607 MiB of VRAM. After loading the model weights and recurrent state:

Total VRAM:                 32,607 MiB
Model weights (Q6_K):       20,409 MiB
RS buffer (GatedDeltaNet):     449 MiB
Compute scratch:               581 MiB
─────────────────────────────────────
Available for KV cache:    ~11,168 MiB

At FP16, the KV cache costs approximately 64 KB per cell (token-position) across 16 attention layers. How far does 11 GB get you?

Configuration KV Cache Size Fits?
FP16, 1 slot x 128K tokens ~8,192 MiB Barely
FP16, 3 slots x 174K (524K total) ~32,768 MiB No. Exceeds total VRAM.
turbo3, 3 slots x 174K (524K total) 7,172 MiB Yes

The actual VRAM breakdown from the running server (from llama-server startup logs):

CUDA0 (RTX 5090): 32,103 MiB total
  Model:    20,409 MiB
  KV cache:  7,172 MiB  (turbo3: K 3,586 MiB + V 3,586 MiB)
  RS state:    449 MiB
  Compute:     581 MiB
  Free:      2,932 MiB

Turbo3 compresses the KV cache by ~4.6x compared to FP16. That’s 3 slots with 3 GB to spare.

How Turbo3 Works

TurboQuant (Zandieh et al., arXiv:2504.19874, ICLR 2026) compresses KV cache values to 3.5 bits per element using a two-step process:

1. Rotate via Fast Walsh-Hadamard Transform (FWHT). Applied to 128-element blocks of KV values. The rotation spreads information uniformly across the block, turning the value distribution into something close to standard normal – ideal for fixed-codebook quantization. Pre/post-multiply with deterministic sign vectors derived from a golden-ratio hash.

2. Quantize via Lloyd-Max codebook. 8 centroids (3 bits) precomputed for the standard normal distribution. Branchless comparison against 7 midpoints gives the centroid index. Each block of 32 values stores as 14 bytes: 12 bytes for the 3-bit indices + 2 bytes for an FP16 block norm. That’s 3.5 bits per value.

On read, dequantization reverses the process: look up centroids, scale by the block norm, apply inverse FWHT. The result feeds into standard attention kernels unchanged.

Measured quality on the KV cache itself: MSE < 0.035, cosine similarity > 0.98.

Key observations:

Decode throughput is consistent: 55.2-55.4 tok/s across analytical and synthesis prompts, regardless of output length. The RTX 5090 is not the bottleneck at short-medium context.

ITL distribution is tight: 18.1 ms mean with 18.5 ms p99 at short context. Sub-millisecond jitter. In practice, it feels smooth.

Tool calling is slower (36.4 tok/s) likely due to the structured output constraint and shorter generation runs where per-token overhead is proportionally larger.

Long context at 131K input tokens degrades to 20.9 tok/s with 47.7 ms ITL mean. This is expected – attention compute is linear in context length per token. At 131K, each decode step touches 131K cached values across 16 attention layers.

Qwen3.5-27B has a thinking mode (on by default) that generates internal reasoning tokens before the visible output. Decode speed is identical – ~55 tok/s regardless. But thinking mode has dramatic effects on quality benchmarks and infrastructure, covered below.

IFEval: The Thinking Mode Finding

Mode Prompt Strict Inst Strict Prompt Loose Inst Loose Eval Time
No-think 88.0% 91.8% 91.3% 94.2% 3,088s
Thinking 28.7% 42.0% 29.6% 42.7% 12,529s

Same model. Same endpoint. Same gen_kwargs. The only difference: the server’s thinking mode was on vs off.

With thinking enabled, IFEval prompt-level strict accuracy drops from 88% to 29%. The model’s internal reasoning tokens (<think>...</think> blocks) leak into the response text that IFEval scores for format compliance. The model hasn’t lost the ability to follow instructions – the thinking tokens break the output format.

Thinking mode hurts IFEval scores and costs 4x the eval time (12,529s vs 3,088s). Disable it for format-sensitive work.

Long-Context Evaluation

RULER: From All Zeros to Near-Perfect

The RULER benchmark initially scored 0.0 across every context length and task type. All zeros. The model appeared completely unable to retrieve, track, or count anything in long context.

The cause was infrastructure, not the model. Qwen3.5’s thinking mode splits output into reasoning_content (the thinking trace) and content (the final answer). Our benchmark harness only read content. That field was empty — the model exhausted its token budget during thinking and never produced a final answer. The model was solving the tasks correctly inside its reasoning trace. Nobody was reading it.

We found the same bug in our production inference Router — the Go service that routes LLM requests for the entire Gnosis agent pipeline. The remote LLM client read content, ignored reasoning_content, and returned empty strings when thinking mode was active. This was likely the cause of empty feed cards we’d been seeing from pipeline agents. The fix was the same everywhere: fall back to reasoning_content when content is empty, strip <think> blocks, and give the model enough token headroom to finish reasoning before answering.

11 of 12 cells passing. The lone holdout is aggregation at 64K — counting city mentions across a 47K-token document requires the model to scan every paragraph. Even 17K tokens of thinking budget wasn’t always enough. The model gets the right answer on 1 of 3 runs; on the other 2 it runs out of reasoning tokens before finishing its count.

Thinking models need token budgets proportional to the work, not the answer. A task that returns a single number can require thousands of tokens of internal reasoning to produce it. If your harness was built for a non-thinking model, your scores are probably wrong. And it’s not just benchmarks — any system consuming thinking model output needs to handle the split response format, or it silently drops the answer.

The pattern is unusual. Perfect at 1-2K. Sharp degradation from 4K to 32K — the model loses needles placed past the 33% depth mark. Then partial recovery at 64-128K.

This NIAH test runs without thinking mode (the password is short enough to extract without reasoning). The U-curve may reflect the hybrid attention/recurrent architecture: the 16 full-attention layers and 48 GatedDeltaNet layers accumulate state differently, and the crossover may create a retrieval dead zone in the mid-range. With thinking mode enabled, the model recovers many of these positions — reasoning lets it compensate for what direct retrieval misses. This deserves deeper investigation, ideally against FP16 KV to isolate whether turbo3 compression is a factor.

What I Would Do Differently

Budget tokens for thinking, not just output. The RULER all-zeros disaster was caused by giving the model 64 output tokens for a task that required thousands of tokens of reasoning. This applies to any thinking model – if your eval harness was designed for non-thinking models, every token budget needs revisiting. We fixed this and went from 0% to 92% (11/12 cells).

Probe the NIAH U-curve. The mid-range degradation (4K-32K) followed by partial recovery (64K-128K) in the custom NIAH benchmark is the most interesting finding in this report and I don’t understand it. It may be an artifact of the hybrid attention/recurrent architecture, or it may be a turbo3 compression artifact. Running the same NIAH against FP16 KV (at lower total context) would isolate the cause.

Test Q8_K quantization. The 7.4-point MMLU-Pro gap from official is significant. Q8_K would use ~24 GB for weights, leaving ~8 GB for KV cache – enough for 1 slot at reasonable context. Worth testing to isolate quantization loss from other factors.

Default to no-think for format-sensitive workloads. The IFEval thinking-mode disaster is a clear lesson. In my production system, I’ve configured no-think as the default for structured extraction tasks and enabled thinking only for research synthesis where the reasoning traces add value.

Consider the 131K decode cliff. Dropping from 55 to 21 tok/s at 131K input is a 2.6x slowdown. Flash Attention with paged KV cache optimizations in newer llama.cpp builds may help, though turbo3 compatibility would need testing.

Reproduction Guide

  1. Hardware: Any GPU with >= 32 GB VRAM (RTX 5090, RTX 4090 with less context headroom, A100). 64 GB system RAM recommended.

  2. Model: Download Qwen3.5-27B-Q6_K.gguf from unsloth/Qwen3.5-27B-GGUF on Hugging Face.

  3. Server: Clone and build the turbo3-cuda fork:

    git clone https://github.com/Madreag/turbo3-cuda.git
cd turbo3-cuda
cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES="89;120"
cmake --build build --config Release -j$(nproc)

  4. Launch:

    ./build/bin/llama-server \
  --model /path/to/Qwen3.5-27B-Q6_K.gguf \
  --n-gpu-layers 99 \
  --ctx-size 524288 \
  --parallel 3 \
  --cache-type-k turbo3 \
  --cache-type-v turbo3 \
  --port 9000 \
  --host 0.0.0.0

  5. Verify:

    curl http://localhost:9000/health
# {"status":"ok"}

  6. Quality benchmarks (requires Python 3.12+, pip install lm_eval):

    lm_eval --model local-chat-completions \
  --model_args model=model.gguf,base_url=http://localhost:9000/v1/chat/completions \
  --tasks mmlu_pro,gsm8k,ifeval \
  --batch_size 1 \
  --output_path ./results

Closing

The headline result is 524K tokens of context at 55 tok/s on a single consumer GPU. But the more useful finding is about thinking models and the infrastructure that serves them.

Thinking models change the contract between the model and everything downstream. The response format splits, token budgets that were generous become starvation-level, and evaluation harnesses that worked for years start returning zeros. We hit all of these in the same week — benchmark scripts, the production inference router, the agent pipeline — and the fixes were trivial once we understood the cause.

Measure your workload, budget your thinking tokens, and read all the fields in the API response.

All benchmark data, scripts, and server configuration are available in the Gnosis v3 repository. The turbo3 compression is from Madreag/turbo3-cuda, implementing the TurboQuant paper by Zandieh et al. (arXiv:2504.19874).
